## 2. Storage Architecture

### Design Principle: Local-First with Optional Cloud Sync

8gent is a CLI tool that must work fully offline. SQLite is the primary store. Convex is an optional sync target for cross-machine persistence.

```
┌──────────────────────────────────────────────────────────────┐
│                    STORAGE LAYERS                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────────────┐     ┌──────────────────────────┐   │
│  │  In-Process Cache   │     │  SQLite (WAL mode)       │   │
│  │  (Working Memory)   │────▶│  .8gent/memory/v2.db     │   │
│  │  Map<string, entry> │     │  ~/.8gent/memory/v2.db   │   │
│  └─────────────────────┘     └──────────┬───────────────┘   │
│                                         │ optional           │
│                                         ▼                    │
│                              ┌──────────────────────────┐   │
│                              │  Convex Cloud Sync       │   │
│                              │  (cross-machine backup)  │   │
│                              └──────────────────────────┘   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Two database files, mirroring the v1 project/global split:

- **Project DB:** `<project>/.8gent/memory/v2.db` — project-scoped memories

## 3. Embedding Strategy

### Model Choice: Ollama nomic-embed-text

| Property | Value |
|----------|-------|
| **Model** | `nomic-embed-text:latest` |
| **Dimensions** | 768 |
| **Context window** | 8192 tokens |
| **Speed** | ~50ms per embedding on M-series Mac |
| **Size** | ~274MB download |
| **Cost** | Free, runs locally |
| **Fallback** | FTS5-only search if Ollama unavailable |

FoodstackOS uses OpenAI `text-embedding-3-small` (1536-dim, cloud). For 8gent, local generation is mandatory to maintain offline operation. `nomic-embed-text` is the best trade-off between quality and resource usage for local inference.

### Embedding Generation

```typescript
interface EmbeddingProvider {
  generate(text: string): Promise<Float32Array>;
  generateBatch(texts: string[]): Promise<Float32Array[]>;
  readonly dimensions: number;
  readonly model: string;
  readonly available: boolean;
}

class OllamaEmbeddingProvider implements EmbeddingProvider {
  readonly dimensions = 768;
  readonly model = "nomic-embed-text:latest";

  get available(): boolean {
    // Check if Ollama is running and model is available
    // Cache result for 30s
  }

  async generate(text: string): Promise<Float32Array> {
    const response = await fetch("http://localhost:11434/api/embeddings", {
      method: "POST",
      body: JSON.stringify({ model: this.model, prompt: text }),
    });
    const { embedding } = await response.json();
    return new Float32Array(embedding);
  }

  async generateBatch(texts: string[]): Promise<Float32Array[]> {
    // Parallel generation with concurrency limit of 4
    return Promise.all(texts.map(t => this.generate(t)));
  }
}

class NullEmbeddingProvider implements EmbeddingProvider {
  // Fallback when Ollama is unavailable
  readonly dimensions = 0;
  readonly model = "none";
  readonly available = false;

  async generate(): Promise<Float32Array> {
    return new Float32Array(0);
  }

  async generateBatch(texts: string[]): Promise<Float32Array[]> {
    return texts.map(() => new Float32Array(0));
  }
}
```

### Embedding Lifecycle

1. **On memory creation:** Generate embedding asynchronously; store memory immediately, embed in background
2. **On memory update:** Re-generate embedding if `content_text` changed
3. **On model change:** Detect model mismatch via `embeddingModel` field; queue re-embedding
4. **Batch operations:** Nightly job to embed any memories missing embeddings

### Storage Format

Embeddings stored as raw `Float32Array` bytes in a BLOB column. For 768 dimensions, each embedding is 3,072 bytes (768 * 4 bytes per float32). 50,000 memories = ~146MB of embedding data.

## 4. Search Architecture

### Hybrid Search Pipeline

Adapts FoodstackOS's hybrid search (`HybridSearchResult`, `HybridSearchOptions`) to SQLite:

```
┌──────────────┐
│  User Query   │
└──────┬───────┘
       │
       ▼
┌──────────────────────────────────────────┐
│           QUERY PROCESSOR                 │
│  1. Tokenize for FTS5                     │
│  2. Generate query embedding              │
│  3. Extract filter criteria               │
└──────┬───────────────┬───────────────────┘
       │               │
       ▼               ▼
┌──────────────┐ ┌──────────────────┐
│  FTS5 Search │ │  Vector Search   │
│  (text match)│ │  (cosine sim)    │
│  rank by BM25│ │  top-K nearest   │
└──────┬───────┘ └──────┬───────────┘
       │               │
       ▼               ▼
┌──────────────────────────────────────────┐
│         RESULT FUSION                     │
│  Reciprocal Rank Fusion (RRF)            │
│  score = Σ 1/(k + rank_i)               │
│  k = 60 (standard RRF constant)         │
│  Weighted: 0.4 * text + 0.6 * vector    │
└──────┬───────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────────┐
│         POST-PROCESSING                   │
│  - Apply importance weighting             │
│  - Apply recency boost                    │
│  - Filter by scope/type/tags              │
│  - Deduplicate                            │
│  - Return top N with attribution          │
└──────────────────────────────────────────┘
```

### Vector Search Implementation

Since SQLite does not have native vector search, we implement brute-force cosine similarity with an optimization path:

```typescript
function vectorSearch(
  db: Database,
  queryEmbedding: Float32Array,
  options: { limit: number; minScore: number; types?: MemoryType[]; scope?: MemoryScope }
): SearchResult[] {
  // Load all embeddings into memory (fine for <100K memories)
  const stmt = db.prepare(`
    SELECT e.memory_id, e.vector, m.importance, m.type, m.scope
    FROM embeddings e
    JOIN memories m ON m.id = e.memory_id
    WHERE m.deleted_at IS NULL
    ${options.types ? `AND m.type IN (${options.types.map(() => '?').join(',')})` : ''}
    ${options.scope ? `AND m.scope = ?` : ''}
  `);

  const results: SearchResult[] = [];
  for (const row of stmt.iterate(...params)) {
    const vector = new Float32Array(row.vector);
    const similarity = cosineSimilarity(queryEmbedding, vector);

    if (similarity >= options.minScore) {
      results.push({
        memoryId: row.memory_id,
        score: similarity * (0.7 + 0.3 * row.importance), // Weight by importance
        matchType: "vector",
      });
    }
  }

  return results.sort((a, b) => b.score - a.score).slice(0, options.limit);
}

function cosineSimilarity(a: Float32Array, b: Float32Array): number {
  let dot = 0, normA = 0, normB = 0;
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i];
    normA += a[i] * a[i];
    normB += b[i] * b[i];
  }
  return dot / (Math.sqrt(normA) * Math.sqrt(normB));
}
```

**Performance note:** Brute-force cosine similarity over 50K memories (768-dim) takes ~15ms on M-series Mac. For >100K memories, we would add an IVF (Inverted File) index or use `sqlite-vss`. This is a Could Have optimization.

### Graceful Degradation

```typescript
async search(query: string, options: SearchOptions): Promise<SearchResult[]> {
  const ftsResults = this.ftsSearch(query, options);

  if (!this.embeddingProvider.available) {
    // Ollama not running — FTS-only mode
    return ftsResults;
  }

  const queryEmbedding = await this.embeddingProvider.generate(query);
  const vectorResults = this.vectorSearch(queryEmbedding, options);

  return this.fuseResults(ftsResults, vectorResults, options);
}
```

Every update creates a version snapshot (mirrors FoodstackOS's audit philosophy):

```typescript
async function updateMemory(
  id: string,
  updates: Partial<MemoryData>,
  reason: string,
  changedBy: string
): Promise<void> {
  const current = await getMemory(id);
  if (!current) throw new Error(`Memory ${id} not found`);

  // Snapshot current state
  db.prepare(`
    INSERT INTO memory_versions (id, memory_id, version, data_snapshot, changed_by, change_reason, created_at)
    VALUES (?, ?, ?, ?, ?, ?, ?)
  `).run(
    generateId("ver"),
    id,
    current.version,
    JSON.stringify(current),
    changedBy,
    reason,
    Date.now()
  );

  // Apply update
  db.prepare(`
    UPDATE memories
    SET data = ?, content_text = ?, version = version + 1, updated_at = ?
    WHERE id = ?
  `).run(JSON.stringify({ ...JSON.parse(current.data), ...updates }), updates.content_text ?? current.content_text, Date.now(), id);
}

async function rollbackMemory(id: string, toVersion: number): Promise<void> {
  const snapshot = db.prepare(
    `SELECT data_snapshot FROM memory_versions WHERE memory_id = ? AND version = ?`
  ).get(id, toVersion);

  if (!snapshot) throw new Error(`Version ${toVersion} not found for memory ${id}`);

  const restored = JSON.parse(snapshot.data_snapshot);
  await updateMemory(id, restored, `rollback to v${toVersion}`, "system");
}
```

### Decay Model

Adapted from FoodstackOS's `decayFactor` and `accessCount` system:

```typescript
/**
 * Calculate effective importance of a memory, accounting for:
 * 1. Base importance (set at creation)
 * 2. Time decay (unused memories fade)
 * 3. Access boost (frequently used memories stay strong)
 * 4. Confidence (for semantic/core types)
 */
function effectiveImportance(memory: MemoryEntry): number {
  const ageMs = Date.now() - memory.createdAt;
  const ageDays = ageMs / (24 * 60 * 60 * 1000);

  // Time decay: half-life of 30 days for unaccessed memories
  const halfLife = 30;
  const timeDecay = Math.pow(0.5, ageDays / halfLife);

  // Access boost: each access extends effective half-life
  const accessBoost = Math.min(1.0, memory.accessCount * 0.05); // Max +50% from accesses

  // Recency of last access matters
  const lastAccessAge = memory.lastAccessed
    ? (Date.now() - memory.lastAccessed) / (24 * 60 * 60 * 1000)
    : ageDays;
  const accessRecency = Math.pow(0.5, lastAccessAge / (halfLife * 2));

  // Base calculation
  let effective = memory.importance * memory.decayFactor;

  // Apply time decay, modulated by access patterns
  effective *= (timeDecay * (1 - accessBoost) + accessRecency * accessBoost);

  // Confidence boost for semantic/core types
  if ("confidence" in memory && memory.confidence) {
    effective *= (0.5 + 0.5 * memory.confidence);
  }

  // Floor: never go below 0.01 (eligible for consolidation/pruning)
  return Math.max(0.01, Math.min(1.0, effective));
}
```

### Consolidation Triggers

```
Effective Importance < 0.1  AND  Age > 7 days   → Candidate for daily consolidation
Effective Importance < 0.05 AND  Age > 30 days   → Candidate for weekly consolidation
Effective Importance < 0.02 AND  Age > 90 days   → Candidate for monthly archetype
Effective Importance < 0.01 AND  Age > 180 days  → Candidate for pruning (soft delete)
```

## 7. Context Window Assembly

The most important function in the memory system. Adapted from FoodstackOS's `getFullContext()` / `ContextWindow` / `ContextWindowArgs`.

```typescript
interface ContextWindowOptions {
  query?: string;                  // Current user message for relevance
  maxTokens?: number;              // Token budget (default: 2000)
  includeLayers?: MemoryType[];    // Which types to include
  scope?: MemoryScope;             // Filter by scope
  minImportance?: number;          // Floor for inclusion
}

interface ContextWindow {
  text: string;                    // Formatted context string for prompt injection
  memories: ContextEntry[];        // Structured entries with attribution
  stats: {
    totalMemories: number;
    byType: Record<MemoryType, number>;
    estimatedTokens: number;
    searchLatencyMs: number;
  };
}

interface ContextEntry {
  memoryId: string;
  type: MemoryType;
  content: string;
  relevanceScore: number;
  importance: number;
  source: string;
}

async function assembleContext(options: ContextWindowOptions): Promise<ContextWindow> {
  const budget = options.maxTokens ?? 2000;
  const entries: ContextEntry[] = [];
  let usedTokens = 0;

  // 1. Working memory first (highest priority, session-specific)
  const working = getWorkingMemories(options);
  for (const w of working) {
    const tokens = estimateTokens(w.value);
    if (usedTokens + tokens > budget * 0.15) break; // Cap at 15% of budget
    entries.push(toContextEntry(w, 1.0)); // Max relevance for working memory
    usedTokens += tokens;
  }

  // 2. Core project knowledge (architecture, conventions)
  const core = await searchCore(options.query, { limit: 10, minImportance: 0.3 });
  for (const c of core) {
    const tokens = estimateTokens(c.summary ?? c.content);
    if (usedTokens + tokens > budget * 0.40) break; // Cap at 40% of budget
    entries.push(toContextEntry(c, c.relevanceScore));
    usedTokens += tokens;
  }

  // 3. Relevant semantic memories (facts, preferences)
  const semantic = await searchSemantic(options.query, { limit: 15, minImportance: 0.2 });
  for (const s of semantic) {
    const tokens = estimateTokens(s.value);
    if (usedTokens + tokens > budget * 0.70) break; // Cap at 70% of budget
    entries.push(toContextEntry(s, s.relevanceScore));
    usedTokens += tokens;
  }

  // 4. Recent episodic memories (what happened recently)
  const episodic = await searchEpisodic(options.query, { limit: 10, minImportance: 0.3 });
  for (const e of episodic) {
    const tokens = estimateTokens(e.summary ?? e.content);
    if (usedTokens + tokens > budget) break; // Fill remaining budget
    entries.push(toContextEntry(e, e.relevanceScore));
    usedTokens += tokens;
  }

  // 5. Format into prompt-ready string
  const text = formatContextWindow(entries);

  return { text, memories: entries, stats: computeStats(entries, usedTokens) };
}
```

async function migrateV1toV2(workingDirectory: string): Promise<MigrationResult> {
  const v1ProjectPath = join(workingDirectory, ".8gent", "memory", "project.jsonl");
  const v1GlobalPath = join(homedir(), ".8gent", "memory", "global.jsonl");

  const result = { imported: 0, skipped: 0, errors: 0 };

  // Read v1 entries
  for (const [path, scope] of [[v1ProjectPath, "project"], [v1GlobalPath, "global"]] as const) {
    const entries = readJsonlV1(path);
    for (const entry of entries) {
      try {
        await storeV2({
          type: classifyV1Entry(entry), // LLM-judge: is this episodic, semantic, or core?
          scope,
          content: entry.fact,
          importance: 0.5, // Default; will be refined
          source: entry.source ?? "import",
          sourceId: entry.id,
          tags: entry.tags,
          createdAt: new Date(entry.createdAt).getTime(),
        });
        result.imported++;
      } catch (err) {
        result.errors++;
      }
    }
  }

  // Keep v1 files as backup (never delete)
  return result;
}

## 12. Technology Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Primary storage** | `bun:sqlite` (WAL mode) | Built into Bun runtime; zero external deps; ACID; concurrent reads |
| **Full-text search** | SQLite FTS5 | Built into SQLite; no separate service; BM25 ranking |
| **Vector storage** | BLOB column in SQLite | Simplest approach; brute-force cosine is fast enough for <100K entries |
| **Embedding model** | Ollama `nomic-embed-text` (768-dim) | Free, local, offline, good quality, small model |
| **Embedding fallback** | FTS5-only search | Graceful degradation when Ollama unavailable |
| **LLM for extraction** | Vercel AI SDK + local/OpenRouter | Follows AI Judging Rule; provider-agnostic |
| **Cloud sync** | Convex (optional) | Already in the stack; eventual consistency |
| **Knowledge graph** | SQLite tables (entities + relationships) | No need for Neo4j/DGraph; graph is small (<10K nodes) |
| **ID generation** | `crypto.randomBytes(6).toString("hex")` with type prefix | Consistent with v1; `mem_`, `ent_`, `rel_`, `ver_` prefixes |
| **Versioning** | Append-only version table | Simple, auditable, rollback-capable |
| **Migration** | Read-only v1 import + LLM classification | Zero data loss; v1 files preserved as backup |

### Why Not...

| Alternative | Why Rejected |
|------------|--------------|
| **PostgreSQL + pgvector** | 8gent is local-first CLI; cannot require a database server |
| **LanceDB** | Extra dependency; SQLite + brute-force cosine is sufficient at our scale |
| **ChromaDB** | Python-based; does not integrate with Bun ecosystem |
| **Pinecone/Weaviate** | Cloud-only; violates local-first principle |
| **FAISS** | C++ native binding complexity; overkill for <100K vectors |
| **Neo4j for graph** | Massive overhead for a small knowledge graph; SQLite adjacency list is fine |
